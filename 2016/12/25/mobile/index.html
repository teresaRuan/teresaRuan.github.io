移动端开发所遇问题小结

Dec 25, 2016 | Hits

  前段时间算是真正意义上参与了一个完整的移动端项目开发，发现实际操作和理论还是有一定的差别的，以下就曾经遇到的问题进行一个小结。

###border-width看起来比实际数值粗
  在实现的过程中，会发现真机上的border比实际设置的数值，比如1px要粗。在此前梳理几个概念：
dpi：与屏幕尺寸及分辨率有关，为每英寸有多少个像素点
dpr(dppx):设备像素比 dpr=设备分辨率/视口大小

对于dpr也可以通过window.devicePixelRatio属性获取。
顾按照这个计算，iphone6的dpr为2，iphone 6 plus的dpr为3。那么当下有一个border属性值为：1px solid #000，实际上，在iphone6上显示的是2px，在6 plus上显示的是3px，这样就造成了在视觉上没有进行还原的情况。可以结合媒体查询进行解决（以下是1px的示例）: 


  @media (-webkit-device-pixel-ratio: 2) {
       .border_test{
        border-width: .5px !important  
        }  
    }
  @media (-webkit-device-pixel-ratio: 3) {
    .border_test{
        border-width: .33333px !important  
        }
}

  这样似乎有几种dpr就要几次使用媒体查询来操作，显得很繁琐。例如手淘就是通过计算设置viewport，使得网页在设备内显示时的页面宽度等于设备逻辑像素大小（device-width），这样在写代码时直接根据设计稿的值计算即可。当scale为1时，device-width=设备物理分辨率/（dpr*scale），dpr和设备物理分辨率都是固定不变的，而设置的就是scale的大小。所以iphone6 的scale为0.5，6plus为0.33333333。
手淘实际截图如下：

  后来在看用chromw看Nexus 5x的时候发现它的dpr为2.6，但是手淘显示的data-dpr是1，感到很奇怪，看了下源码，其实际计算dpr的代码如下 

if (!dpr && !scale) {
        var isAndroid = win.navigator.appVersion.match(/android/gi);
        var isIPhone = win.navigator.appVersion.match(/iphone/gi);
        var devicePixelRatio = win.devicePixelRatio;
        if (isIPhone) {
            // iOS下，对于2和3的屏，用2倍的方案，其余的用1倍方案
            if (devicePixelRatio >= 3 && (!dpr || dpr >= 3)) {                
                dpr = 3;
            } else if (devicePixelRatio >= 2 && (!dpr || dpr >= 2)){
                dpr = 2;
            } else {
                dpr = 1;
            }
        } else {
            // 其他设备下，仍旧使用1倍的方案
            dpr = 1;
        }
        scale = 1 / dpr;
    }

zepto ios 9 ，safari input事件无效

现象： 我们用zepto注册事件的格式都是$ele.on([事件类型],[执行函数])。但发现在ios9的系统的safari浏览器中写如下代码是无效的: 

$ele.on( 'input', function() {
    //事件不触发
    ...
} )

这应该是属于zepto在事件注册实现上的问题，jquery是没有问题的。
解决方案： 

$ele.on( 'input propertyChange', function() {
    //事件触发
    ...
} )

safari在触屏滑动时会有卡顿现象

现象: 在iphone的safair（ios9 其它未测）中有时触屏滑动时会有卡顿的现象
解决方案： 添加-webkit-overflow-scrolling：touch。这个属性可以做到快速滚动和回弹效果。safari是用了原生控件来做了这个实现。

这个属性是真的创建了带有硬件加速的系统级控件，效率高，但相对耗费更多的内存。

ps：移动端出了一个新特性Passive Event Listeners，提供页面滑动流畅度，事件监听中的添加一个属性，继续学习继续学习。

Android下UC浏览器表单提交响应速度慢

问题： 在UC中假设一个登录结构如下： 

<input type="text" name="username" />  
<input type="password" name="password" />  
<a href="javascript:">登录</a>

它会在页面加载时检测到有密码类型的输入框存在即将这个作为表单处理，因此会造成一定时间的等待。
解决方法： 在一开始把密码输入框的type写为text在页面加载完成后将其用js改为password。
注：在提交的时候用了a标签，在处理的时候记得添加处理，避免用户多次重复点击造成多次请求接口（自我提醒）
